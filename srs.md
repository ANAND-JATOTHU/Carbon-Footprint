Software Requirements Specification (SRS)1. System Architecture & EnvironmentSince you are a solo developer competing in a time-bound challenge, the system uses a monolithic, single-container architecture optimized for fast deployment on Google Cloud Run.Frontend Client: React.js (Static build served by the backend).Backend Server: Python with FastAPI (chosen for async performance and built-in OpenAPI documentation, which looks highly professional).Database: PostgreSQL (Relational structure is best for leaderboards and user tracking).Hosting: Google Cloud Run (Dockerized, serverless, scales to zero).2. Functional Requirements (Core System Features)2.1. Authentication & Session ManagementFR-1: The system shall authenticate users via standard Email/Password.FR-2: Upon successful login, the system shall issue an HTTP-only, short-lived JSON Web Token (JWT) valid for 24 hours.FR-3: The system shall restrict access to the /api/actions and /api/profile endpoints to requests containing a valid JWT.2.2. Carbon Calculation EngineFR-4: The system shall ingest integer and string values from the onboarding form (e.g., miles_driven, diet_type).FR-5: The system shall multiply user inputs by standardized emission factors (stored as environmental variables or a static backend dictionary) to calculate the baseline metric tons of CO2.FR-6: The system shall update the user's total score in real-time when a new "action" is logged.2.3. Anti-Cheat & Gamification (The Leaderboard)FR-7: The system shall aggregate user scores and expose a public /api/leaderboard endpoint sorted by the lowest carbon footprint.FR-8: The system shall strip all Personally Identifiable Information (PII), returning only randomized usernames (e.g., "EcoWarrior_99") and scores to the frontend leaderboard.3. Non-Functional Requirements3.1. Security & Privacy (The "Winning Edge")NFR-1 (Rate Limiting): The /api/actions/submit endpoint shall be rate-limited using a sliding window algorithm to a maximum of 5 requests per minute per IP address to prevent script-based leaderboard manipulation.NFR-2 (Anomaly Detection): The backend shall reject any logged action that claims a CO2 reduction of greater than 50kg in a single payload.NFR-3 (Container Hardening): The Dockerfile must explicitly create and switch to a non-root user (USER appuser) before executing the FastAPI server.3.2. Performance & ReliabilityNFR-4: API response times for calculating the footprint and retrieving the leaderboard must remain under 200 milliseconds.NFR-5: The application must be entirely stateless in the backend. All session data must be in the JWT, and all persistent data in PostgreSQL, ensuring Cloud Run can spin up multiple instances if traffic spikes.4.
