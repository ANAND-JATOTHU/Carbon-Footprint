Product Requirements Document (PRD)1. Executive Summary & Problem Statement1.1 ObjectiveTo design and build a secure, lightweight, web-based Carbon Footprint Awareness Platform that empowers individuals to calculate their baseline emissions, track reduction efforts through verified micro-habits, and engage in safe, gamified community leaderboards without compromising personal privacy.1.2 The ProblemExisting carbon calculators suffer from three distinct barriers to user adoption:High Friction / Low Engagement: They require exhaustive manual data entry (e.g., utility bill serial numbers, precise vehicle mileage history), causing high drop-off rates during onboarding.Privacy Risks: Users are forced to surrender highly personal behavioral patterns (dietary restrictions, exact commuting routines, geographic location) to central servers without transparent data protection.Leaderboard Exploitation: Existing gamified platforms rely on unvalidated user self-reporting, allowing malicious actions (e.g., automated API scripting) to easily manipulate global rankings, destroying community trust.1.3 The Solution (The Competitive Edge)A single-page application built on a stateless Python/FastAPI backend and React frontend that delivers an instant, heuristic 60-second baseline assessment. The platform uses zero-knowledge architectural principles—keeping granular behavioral choices local to the client while transmitting only aggregated carbon weights to a hardened, rate-limited, pseudonymized leaderboard.2. User PersonasPersona A: The Eco-Conscious Commuter (Target User)Demographics: Working professional, mobile-first web user.Goals: Wants to understand their relative impact on climate change and implement simple daily changes without adding friction to an already busy schedule.Pain Points: Highly skeptical of platforms asking for explicit tracking permissions or linking bank accounts; easily overwhelmed by generic "stop flying entirely" mandates.Persona B: The Hackathon Judge / Security Auditor (The Evaluation Persona)Demographics: Senior Technical Engineer, Security Lead.Goals: Validates the software architecture under stress, verifies that authentication mechanisms are robust, and actively tests for systemic logic flaws or injection vulnerabilities.Pain Points: Frustrated by simple CRUD boilerplate applications that display mock data; actively looks for validation bypasses and missing server-side sanitization.3. Scope & Feature RequirementsThe product scope is strictly contained to an MVP capable of immediate single-container deployment.3.1 Epic 1: Low-Friction Onboarding & Heuristic EstimationFR-1.1 (Macro-Category Form): The system must present a 3-stage onboarding wizard capturing structural habits across three vectors: Transport (Primary vehicle type + distance ranges), Diet (Frequency of animal-protein intake), and Home (Housing square-footage tier + energy grid archetype).FR-1.2 (Client-Side State Storage): Granular onboarding parameters must be cached directly within browser storage (localStorage) to preserve session state across page reloads without database overhead.FR-1.3 (Heuristic Math Engine): The application must process inputs using deterministic conversion matrices derived from EPA and IPCC greenhouse gas equivalents to calculate annual metric tons of $CO_2$ equivalents ($MTCO_2e$).3.2 Epic 2: Gamified Engagement & Anti-Cheat LeaderboardsFR-2.1 (The Micro-Habit Ledger): The dashboard must serve a randomized rotation of achievable daily activities (e.g., "Line-dry laundry", "Switch to public transit for today's commute") mapped to verified reduction weights.FR-2.2 (The Pseudonymized Global Standings): The platform must display a real-time leaderboard aggregation showing the top-performing reduction metrics globally.FR-2.3 (Privacy-Preserving Display Names): The system must automatically map account credentials to randomized, non-identifiable display names (e.g., SolarPanda_4021) on public routing views to fully anonymize internal analytics.3.3 Epic 3: Enterprise Threat Mitigations (Leaderboard Security)FR-3.1 (Sliding-Window Rate Limiting): All mutation endpoints (/api/actions/submit) must evaluate request velocities and automatically reject actions exceeding 5 requests per minute from a distinct IP address.FR-3.2 (Payload Anomaly Filtering): The backend validation pipeline must enforce logical boundaries on reduction tokens (e.g., rejecting any individual action declaration that claims a reduction exceeding 50kg $CO_2$ in a single transaction).FR-3.3 (Live Telemetry Dashboard): An administrative sub-view must expose active middleware actions, tracking rate-limiting triggers and authorization rejections live to visualize the application defending itself against structural manipulation.4. Non-Functional Requirements & Constraints4.1 Security & Data IntegrityNFR-4.1: Session state management must run exclusively via cryptographic JWT payloads stored in HTTP-Only, Secure cookies to eliminate Cross-Site Scripting (XSS) token harvesting vectors.NFR-4.2: Password structures must be run through high-iteration salt hashing algorithms (e.g., bcrypt or argon2) before entering persistence fields.4.2 Performance & Operational ScaleNFR-4.3: The calculation middleware layer must run asynchronously, processing incoming telemetry payloads and delivering state response packets in $<200\text{ms}$.NFR-4.4: The deployment framework must adhere to a fully stateless structural model, decoupling instance execution context from the underlying file system to allow instant horizontal scaling on Google Cloud Run.5.
